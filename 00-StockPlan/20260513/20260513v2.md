## 5. 目前結論

目前系統的最大阻塞 **不是程式碼能力，也不是 PIT gate / validator / coverage preview 不足**，而是 **缺少真實 TWSE historical MonthlyRevenue source files**。

從 P26F3 開始已確認：本地無 2025-09～2026-01 月營收真實 source；template rows 不算 real coverage；必須人工取得 TWSE 歷史資料後才能往 P26F4 controlled import gate 前進。

P26F3-4 已把這個人工取得流程整理成 package，因此目前 roadmap 應停止繼續補內部工具，改為執行 **資料取得 / 放檔 / 驗證 / coverage preview**。

# CTO Agent 分析：下一階段最值得優化方向

## Roadmap 對齊度檢查

### 目前 roadmap 實際狀態

P26A～P26F3-4 的路徑是合理的：

1. 先修 MonthlyRevenue PIT contract / reason context。
2. 再確認 NewsEvent / FinancialReport 只能 read-only。
3. 再做 coverage comparison。
4. 再定位 MonthlyRevenue coverage=0。
5. 再定位 releaseDate / historical source 缺口。
6. 最後建立 manual acquisition package。

目前與 roadmap 的差距在於：**P26F4 不能自動啟動**。P26F4 的前置條件是 acceptedRows > 0、matchedRows > 0、safety PASS、scoring invariance PASS、CTO approval。現在 acceptedRows=0、matchedRows=0，所以直接進 P26F4 會違反 gate。

### Roadmap 必須調整

原本「P26F4 Controlled Import Gate」應下修為 **conditional stage**，中間插入一個實作性階段：

> **P0 / Immediate：Manual TWSE Data Acquisition Execution**

也就是先完成真實資料取得與 validator pass，再進 import gate。

---

# 最新 Roadmap：P0～P10

## P0 — Manual TWSE Historical MonthlyRevenue Data Acquisition Execution

**目標**：取得 2025-09～2026-01、25 symbols 的 TWSE / MOPS 月營收檔案，放入 drop-zone，跑 validator。

**Gate**：

* candidateFiles > 0
* acceptedRows > 0
* rejectedRows 可解釋
* matchedRows > 0
* safety PASS
* scoring invariance PASS

**原因**：這是目前唯一真正阻塞 P26F4 的缺口。

---

## P1 — P26F4 Controlled Import Gate

**目標**：只在 P0 通過後，建立 controlled import gate。

**內容**：

* dry-run import plan
* migration readiness
* row diff preview
* duplicate handling
* rollback plan
* CTO approval token gate

**禁止**：沒有 CTO approval 不得 DB write。

---

## P2 — Fix Pre-existing TSC Error

**目標**：修復 `src/app/api/admin/data-quality/route.ts` 的既有 TSC error。

**原因**：目前所有 P26 tasks 都必須補一句「TSC error pre-existing」。這會降低後續 PR / CI 判斷品質。

**Gate**：

* `npx tsc --noEmit` clean
* 不影響 P26 artifacts
* 不改 scoring

---

## P3 — Formal Historical MonthlyRevenue Import

**前置**：P1 approval 通過。

**目標**：

* Apply required migration。
* Import accepted MonthlyRevenue historical rows。
* releaseDate must be verified。
* DB diff report / rollback plan。

---

## P4 — Re-run P26F / Candidate Corpus Coverage

**目標**：DB 有 real historical rows 後，重新跑 P26F candidate corpus dry-run。

**Gate**：

* candidate matchedRows > 0
* alphaScore / recommendationBucket 0 mismatch
* original corpus unchanged

---

## P5 — Formal Corpus Expansion PR Gate

**目標**：只在 P4 coverage > 0 且 invariance pass 後，才建立 corpus expansion PR。

**注意**：這仍不是 optimizer，也不是 performance evaluation。

---

## P6 — NewsEvent / FinancialReport Source Mapping Revisit

**目標**：目前 NewsEvent / FinancialReport 都是 fixture-only，不應進 scoring。待 MonthlyRevenue 路線解完後，才回來做 source mapping。

---

## P7 — PIT-safe Feature Context Consolidation

**目標**：把 MonthlyRevenue / NewsEvent / FinancialReport context 的 PIT gate 規格統一成一份 machine-readable feature availability registry。

---

## P8 — Backtest / Replay Observability Upgrade

**目標**：加強 coverage、missing reason、availability reason、source provenance 的 replay observability。

---

## P9 — Strategy / Ranking Layer Readiness Review

**前置**：資料 coverage 有實質改善後才做。

**禁止**：不得宣稱 alpha / edge / ROI。

---

## P10 — Controlled Optimizer Exploration Gate

**前置**：

* P3 import 完成
* P4 candidate coverage > 0
* P5 corpus expansion accepted
* P8 observability 足夠

**定位**：paper-only / observability-only，不得 production trading。
