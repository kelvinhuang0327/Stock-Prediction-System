# Stock Prediction System — 每日工程交接報告

## 1. 本輪目標

本輪主要目標是依據《股票投資輔助工具上線驗證分析報告》的建議，將開發方向從單純延伸 walk-forward skeleton，調整為 **MVP 上線驗證前的資料安全與 shadow validation 基礎建設**。

核心工作分成兩條主線：

1. **上線驗證 P0 Guardrail**

   * 防止 future-date leakage。
   * 建立 as-of data gate。
   * 確保 MVP API 不使用 `asOfDate` 之後資料。
   * 補齊 MarketIndex / MarketRegime 的 as-of gate。

2. **Shadow Prediction Log Contract**

   * 建立每日研究候選分數與理由的 shadow log contract。
   * 僅做 research-mode / dry-run / append-only contract。
   * 不寫 production Prediction rows。
   * 不產生交易建議。
   * 不宣稱策略績效。

## 7. 風險與不確定點

### 1. DB 仍保留 future rows

目前做法是 query / API 層排除 future rows，但 DB 內仍可能存在：

```text
date > asOfDate
```

風險：

* 未被 gate 的舊路徑仍可能誤讀。
* 新功能若未沿用 AsOfDataGate，可能再次引入 leakage。

建議：

* 後續建立 global query audit。
* 建立 future-row quarantine dashboard。
* 非必要不直接刪 DB，先保守隔離。

---

### 2. Shadow log 目前只是 contract

P0-02B 只完成 contract，不是正式 daily writer。

風險：

* 還沒有實際每日 ledger。
* 還沒有 append-only guard 實際落地。
* 還沒有 run schedule。
* 還沒有 outcome 回填。

---

### 3. researchScore / researchBucket 容易被誤解

目前已將：

```text
alphaScore -> researchScore
recommendationBucket -> researchBucket
```

但產品或前端若顯示不當，仍可能被使用者理解成投資建議。

建議：

* 前端需加研究模式 disclaimer。
* 不使用「推薦」、「買進」、「賣出」、「勝率」等語意。
* 顯示 limitations / missing sources / data coverage。

---

### 4. `sourceDate <= asOfDate` 只是初步 PIT 防線

它能防止明顯未來資料，但不能完全保證：

* 發布時間可得性
* T+1 availability
* 收盤後才可取得的資料是否被當日使用
* derived feature 是否偷看未來

後續仍需 PIT audit。

---

### 5. MarketRegime out-of-scope callers 尚未完全關閉

P0-04 主流程完成，但還有 7 個 caller 未處理。

如果這些 caller 會出現在 MVP API 或 Daily Ops，就需要補 gate。
